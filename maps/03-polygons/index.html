Ideation Phase

Now into my 3rd day of Me + AI mapping, I asked the 3 AI models to come up with Polygons map ideas. The first ideas they gave me were not bad, but basic. Mostly geographical boundaries like country or state border outlines and filling them with colors based on data. So I workshoped with AI to rewrite the prompt around more impactful polygons that don't get mapped often, and retried with more success - the second round of ideas from the LLMs were better! Ideas like noise "petals", watershed "mosaics", and tree canopy maps. Now that's worth trying.

Where I Redirected The Prompt
  • Called out the first batch for boring shapes like state outlines and asked for more interesting shapes with creativity or human impact.
  • Added constraints about interesting boundaries plus lightweight, embeddable datasets.
  • Selected the airport-noise petals once I verified the FAA/BTAA data was readily downloadable.

Chosen Concept

Airport Noise Footprint (PDX)

Day-Night Average Sound Level (DNL) contours around Portland International Airport, rendered as translucent lobes that mirror the flight paths slicing across northeast Portland.

Data Source: PDX Part 150 Noise Exposure Map contours via BTAA Geoportal (with synthetic fallback if download fails)

Why chosen: Irregular, high-impact polygons that make an invisible quality-of-life boundary visible while keeping the dataset small enough for every platform to attempt.

Implementation Prompt

Used the same prompt across all platforms:

PROJECT: #30DayMapChallenge 2025 - Day 3 (POLYGONS)

CONCEPT:
Airport Noise Footprint (PDX). Render the 60/65/70 dB Day-Night Average Sound Level (DNL) contours as translucent, patterned petals that radiate from Portland International Airport.

LOCATION:
PDX (45.5898¬∞ N, -122.5951¬∞ W) with ~10‚Äì15 km radius coverage.

VISUAL REQUIREMENTS:
- Interactive web map with modern, responsive layout.
- Each contour band gets a distinct fill (pattern/gradient) with opacity 0.3‚Äì0.6 and thin outline.
- Include title, legend, attribution, and optional toggles for each DNL band.

DATA FLOW:
1. Try downloading the PDX Part 150 contours from https://geo.btaa.org/catalog/e6b6fe6c6a6840a096f89f6be703dc64_144.
2. If that fails, attempt the City of Portland overlay guide (PDF/GIS) at https://www.portland.gov/ppd/.../download.
3. If both fail, synthesize 3‚Äì4 irregular polygons centered on PDX with dnl values [70, 65, 60] and believable flight-path lobes.

IMPLEMENTATION NOTES:
- Convert shapefiles to GeoJSON, clip to the study area, and simplify < 100 KB.
- Final GeoJSON must expose properties.dnl (number) and properties.level (label).
- Use basemaps that do not require API keys (Leaflet + OSM, MapLibre, CartoDB, etc.).
- Tooltips: show dnl level, short description, and reminder that DNL is 24-hour averaged noise.
- Optional: checkbox toggles per contour and an airport marker.

DELIVERABLE:
One-page HTML/JS map that runs without a build step.

Key Insights

‚úÖ What Worked
  • Firebase Studio finally pulled the real PDX Part 150 contours: first genuine data ingestion of the challenge.
  • MagicPatterns, Bolt.new, and Base44 all fell back to synthetic petals gracefully while keeping UI polish high.
  • Cross-platform legends and toggles got better; 3 partial builds generated usable layer control without prompting.
‚ùå What Didn't
  • Mapbox token prompts stopped Lovable despite valid creds: API keys remain a blocker for rapid testing.
  • Claude and Gemini still hit the same CSP/tile restrictions from Days 1‚Äì2, leaving maps blank.
  • ChatGPT Canvas refused to fetch the real data even after multiple retries, sticking with mock GeoJSON.
üéâ Surprises
  • Firebase's browser IDE was clunky to prep, but the payoff (real data on-screen) made the API-key dance worth it.
  • Base44 politely offered an upload slot for cleaned GeoJSON, hinting at an easy handoff when we pre-process later.
  • Google AI Studio moved from total failure to at least rendering the shell, even if the map still never loads.
üìö Lessons for Next Time
  • Include ready-to-embed GeoJSON for the next run so sandboxed platforms can still prove rendering chops.
  • Keep MapLibre/OSM as the default callout in every prompt to avoid Mapbox blocking issues.
  • Document data-handling expectations early: platforms default to mock data unless directly asked for real data.

Editorial

Day 3 felt like the inflection point: the concept required every platform to grapple with real GIS workflows instead of canned CSVs. Watching Firebase Studio grind through the API-key setup and still emerge with live PDX contours was the highlight: it proves the pipeline can work when the tooling cooperates. At the same time, the consistency of the partials tells its own story: if a platform leaned on mock data on Day 1 or got blocked by CSP on Day 2, it behaved the exact same way here. Those patterns are now "known unknowns" we can design around.

For future days, I'm gonna get the data to work better. Maybe upload it myself, so even the sandboxed tools can show what they would do with the real data. I'm also rewriting the base prompt to not use Mapbox: no more getting stuck on token dialogs. The experiment continues to validate the parallel approach: multiple working prototypes in ~60 minutes, plus a clearer picture of where the human needs to guide.
