Implementation Prompt

Used the same prompt across all platforms:

PROJECT: #30DayMapChallenge 2025 - Day 2: LINES

CONCEPT:
Create an interactive "Street Orientation Rainbow" map of Portland, Oregon. Color-code every street segment by its compass bearing (0-360¬∞) using a full HSL color wheel, so cardinal directions appear as distinct colors. This creates a visual fingerprint of Portland's urban form - revealing the city's iconic grid system alongside its more organic neighborhoods.

REQUIREMENTS:

Visual:
- Interactive web-based map centered on Portland
- Each street segment colored by its compass bearing using HSL color scale:
  - 0¬∞ (North) = Red
  - 90¬∞ (East) = Yellow/Green
  - 180¬∞ (South) = Cyan
  - 270¬∞ (West) = Purple/Magenta
- Line thickness varies by road type (motorway/trunk thickest, service roads thinnest)
- Dark basemap that makes colored lines stand out
- Modern, clean design with clear visual hierarchy
- Responsive layout (works on mobile and desktop)
- Legend showing the color-to-direction mapping (circular color wheel preferred for directional data)

Data:
- Data source: OpenStreetMap via Overpass API (free, no API key required)
- Endpoint: https://overpass-api.de/api/interpreter
- Portland bounding box: 45.43,-122.84,45.65,-122.47
- Query for streets: way["highway"]["highway"~"motorway|trunk|primary|secondary|tertiary|residential|unclassified|living_street|service"]
- Returns street geometries with highway type and names

Functionality:
- Zoom and pan controls
- Hover tooltips showing: street name, bearing (0-360¬∞), road type
- Smooth interactions (no excessive lag when panning/zooming)
- Optional: Click to show more details in a popup

Technical:
- Calculate bearing/angle for each street segment from its coordinates
  - Formula: bearing = Math.atan2(lon2-lon1, lat2-lat1) * (180/Math.PI)
  - Normalize to 0-360 range
  - Convert bearing to HSL color: hsl(bearing, 85%, 60%)
- Use modern JavaScript (ES6+)
- Mapping library: Your choice (Leaflet, Mapbox GL JS, MapLibre GL JS, deck.gl, or platform's preferred library)
- Fetch and parse Overpass API JSON response
- Handle the data and render lines efficiently
- Use CDN links for any external libraries
- No build step required - should run directly

DELIVERABLES:
- Fully functional interactive street orientation rainbow map
- Title: "Street Orientation Rainbow: Portland, Oregon"
- Attribution: "Data: ¬© OpenStreetMap contributors"
- Legend showing color-to-direction mapping and road type hierarchy
- Brief description: "Portland's streets colored by compass bearing. Each color represents a different direction, revealing the city's grid patterns and organic growth."
- Clean, commented code
- Responsive and works in modern browsers

Editorial

Day 2 worked, but revealed how data size and complexity separate robust platforms from fragile ones. The 91MB Overpass API dataset challenged performance limits, and geometry calculations (bearing from coordinates) tested platform capabilities in new ways.

The Winner: MagicPatterns is the clear winner for Day 2 - fastest data loading, best pan/zoom performance (~1-2 sec lag vs. ~10 sec for others), most intuitive circular legend, and strong mobile UX. Zero human intervention needed. Superior out-of-the-box quality in every category. Not a household name, but delivered the best overall result.

The Big Challenge: The 91MB dataset exposed platform weaknesses. ChatGPT Canvas took ~1 minute to render data (initially appeared broken). Bolt.new and ChatGPT Canvas both had sluggish pan/zoom (~10 sec lag). Firebase Studio's autonomous debugging identified the Next.js 2MB cache limit issue but couldn't resolve it. Only MagicPatterns and Lovable (both using MapLibre GL) handled the large dataset smoothly.

The Legend Insight: Circular legends (Bolt.new, MagicPatterns) were far more intuitive than linear gradients or 4-square compass roses for directional data. This makes sense - compass bearings are cyclical (0¬∞ = 360¬∞), and a circular color wheel visually represents that relationship. For future days with directional data, circular legends should be directly requested.

The Mobile Reality Check: Base44's non-collapsible mobile legend taking >50% of the screen shows how critical mobile UX is. Despite solid technical execution, the map was nearly unusable on phones. MagicPatterns' mobile experience was strong - another reason it won Day 2.

The Patterns: Consistent performers: Lovable, Bolt.new (Day 1 + Day 2 both successful). Consistent partial successes: Claude Artifacts, Gemini Canvas, Firebase Studio ‚Äî they build UI but sandbox limits keep them from full delivery. Unreliable: Google AI Studio (Day 1 worked, Day 2 error). Dark horse: MagicPatterns (delivered #1 result despite being lesser-known).

Day 1 vs Day 2: Day 1 had 73% success rate with a ~500KB dataset. Day 2 had 30% full success (30% truly usable) with a 91MB dataset. Partial success rate is 60% (6 platforms) vs. Day 1's 27% (3 platforms), but those runs still need manual fixes before they're shareable. One complete failure (Google AI Studio). The conclusion is clear: larger datasets + geometry calculations created a bigger challenge, separating robust platforms from fragile ones. Day 2 was more discriminating in identifying truly capable platforms.

Looking ahead: Day 2 established that performance matters - not just "does it work" but "does it work well". For future days with large datasets or complex calculations, MagicPatterns and Lovable (MapLibre GL) are the top choices. For smaller datasets, the broader field remains viable. But the real lesson: data size and complexity reveal platform robustness in ways that simpler tests don't.

MagicPatterns wins Day 2, but the bigger win is understanding which platforms handle real-world complexity.
